Dərs 04-A — Sensor Sisteminin Fiziki Qurulması, Təchizatı və Məsuliyyət Zənciri

Bu dərsə qədər biz siqnalları Python-da simulyasiya edirdik. İndi eyni siqnalların Siyəzən yem zavodunda real cihazlardan necə yarandığını, kimin quraşdırdığını, hansı yolla mərkəzə ötürüldüyünü və nəhayət bizim kompüterdə istifadəyə necə hazırlandığını izah edirik. Bu, developer işi deyil — bunu müəssisə müdüriyyəti sifariş verib icra etdirməli və developer işə başlayana qədər hazır olmalıdır.

5 sensor növü Sensordan dashboard-a 10 mərhələli məsuliyyət zənciri Təchizat qrafiki Büdcə və bazar qiymətləri

01 — Kontekst

Simulyasiyadan reallığa: niyə ayrıca mərhələ?

Dərs 03 və 04-də sensor_simulyator.py təsadüfi, amma məntiqli ədədlər yaradıб MQTT-yə göndərirdi. Bu, sistemi proqram tərəfdən qurmaq və sınamaq üçün idealdır — real avadanlıq olmadan bütün zənciri (broker → FastAPI → edge baza → dashboard) işlək hala gətirdik.

Lakin real zavodda o ədədlər simulyatordan yox, fiziki cihazlardan gəlməlidir. Bu keçid təsadüfi olmur: cihazların alınması, düzgün yerə quraşdırılması, elektrik və şəbəkə işləri, kalibrləmə və qəbul — hamısı ayrıca bir təchizat və montaj layihəsidir. Bu işlər ay(lar) çəkə bilər, ona görə müdüriyyət bunu indidən planlaşdırmalıdır ki, developer işə başlayanda infrastruktur hazır olsun.

Əsas prinsip Developer (siz) heç bir sensor quraşdırmır, kabel çəkmir. Sizin işiniz — hazır MQTT axınını qəbul edən proqram təminatıdır. Bu dərs məhz o axını yaradan tərəfi — müdüriyyət, mühəndislər və podratçıları — təsvir edir.

03 — Avadanlıq

Alınacaq cihazların texniki növləri

Müdüriyyət və təchizat şöbəsi üçün ən vacib sual: konkret nə almaq? Aşağıdakı cədvəl hər sensor növü üçün sənaye standartı avadanlığı və çıxış siqnalını göstərir. Siqnal standartı vacibdir — bütün cihazlar ya 4-20 mA analoq, ya da Modbus RTU (RS-485) rəqəmsal çıxış verməlidir ki, gateway onları oxuya bilsin.

04 — Siqnal Axını

Sensordan mərkəzə: siqnal necə gedir?

Bir temperatur oxumasının fiziki zonddan bizim dashboard-a qədər keçdiyi yol yeddi mərhələdən ibarətdir. Bu zəncirin ilk dörd mərhələsi zavodda (fiziki), son üçü isə bizim proqram tərəfindədir (Dərs 04–07-də qurduğumuz).

1–2. Fiziki siqnal və transmitter

Sensorun özü zəif/xam elektrik siqnalı verir. Transmitter onu standart 4-20 mA cərəyan siqnalına və ya rəqəmsal Modbus RTU mesajına çevirir. 4-20 mA sənayedə üstünlük təşkil edir, çünki uzun kabellərdə maneədən təsirlənmir (4 mA = minimum, 20 mA = maksimum dəyər).

3. Gateway / PLC — zəncirin ürəyi

Bu cihaz bütün sensorları oxuyur (analoq girişlər + Modbus şini) və onları MQTT mesajlarına çevirib brokerə göndərir. Topic strukturu bizim .env-dəki MQTT_TOPIC_BASE=zavod/siyezen ilə uyğun olmalıdır:

# Hər cihaz öz topic-inə yazır — cihaz kodu ilə:
zavod/siyezen/S001   # temperatur
zavod/siyezen/S002   # rütubət
zavod/siyezen/S003   # çəki
zavod/siyezen/S004   # vibrasiya
zavod/siyezen/S005   # enerji

# Mesajın formatı (JSON) — FastAPI-nin gözlədiyi ilə eyni:
{ "cihaz_kod": "S001", "qiymet": 24.3, "olcme_vaxti": "2026-07-11T09:15:00+04:00" }
Kritik uyğunluq Gateway-in göndərdiyi topic adları və JSON sahələri bizim FastAPI-nin (Dərs 05) gözlədiyi ilə eyni olmalıdır. Bu, developer ilə inteqrator arasında razılaşdırılan "müqavilə"dir. Uyğunsuzluq olsa, məlumat baza yerinə itir.

4. MQTT Broker (Mosquitto)

Broker "poçt mərkəzi"dir: cihazların göndərdiyi mesajları qəbul edir və onlara abunə olan FastAPI-yə çatdırır. Zavodda broker ya lokal edge serverdə (bizim maşın rolunda), ya da müəssisə şəbəkəsində işləyir. Bağlantını yoxlamaq üçün sadə əmr:

# Bütün zavod topic-lərini canlı dinlə (test üçün):
mosquitto_sub -h localhost -p 1883 -t "zavod/siyezen/#" -v

08 — Nəzarət

Developer işə başlamazdan əvvəl hazır olmalı olanlar

Bu siyahının hər bəndi təsdiqlənəndə, sistem developerə təhvil verilməyə hazırdır:

  • Beş sensor (S001–S005) düzgün zonalara quraşdırılıб və kalibrlənib.
  • Gateway real ölçmələri zavod/siyezen/* topic-lərinə göndərir.
  • MQTT broker (Mosquitto) əlçatandır və mosquitto_sub ilə canlı data görünür.
  • Göndərilən JSON formatı və topic adları TZ ilə (FastAPI-nin gözlədiyi ilə) eynidir.
  • Şəbəkə stabil, təhlükəsizlik (parol/VPN) qurulub.
  • Bir neçə saatlıq nümunə data yığılıb və dəyərlərin məntiqli olduğu yoxlanılıb.
  • Təhvil aktı imzalanıb, zəmanət və xidmət müqaviləsi var.
Növbəti addım Bu siyahı təsdiqlənənə qədər biz simulyator (Dərs 04) üzərində işləməyə davam edirik — real infrastruktur hazır olan kimi simulyatoru gateway ilə əvəz edəcəyik. Bu dərs həmin keçidin təşkilati "yol xəritəsidir".
