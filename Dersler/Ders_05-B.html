02 — Texniki maneə

Sap problemi: paho sinxrondur, FastAPI asinxron

Birləşdirmə bir texniki maneəyə dirənir və bunu başa düşmək kritikdir.

Problem paho-mqtt sinxron kitabxanadır — o, öz ayrı sapında (thread) işləyir. on_message callback-i məhz o sapdan çağırılır.

FastAPI/asyncio isə tək saplı — bütün async kod bir "event loop"-da işləyir.

Nəticə: on_message içindən birbaşa await hovuz.connection() yaza bilmərik — o, async funksiyadır və başqa sapdadır. Sadəcə await yazmaq RuntimeError: no running event loop verər.

Həll: növbə (queue) ilə körpü

Klassik və etibarlı yanaşma: MQTT sapı mesajı növbəyə atır, async tapşırıq isə növbədən oxuyub bazaya yazır. İki dünya arasında təhlükəsiz keçid.

Sap sərhədini keçmək
paho sapıon_message
►
asyncio.Queuetəhlükəsiz körpü
►
async yazıcıevent loop
►
Edge DBbatch INSERT
Niyə bu, sadəcə "texniki hiylə" deyil — memarlıq qazancıdır Növbə bizə üç pulsuz hədiyyə verir:

① Bufer. Baza bir anlıq yavaşlasa, mesajlar növbədə gözləyir — MQTT sapı bloklanmır, mesaj itmir.
② Batch imkanı. Növbədən bir-bir yox, dəstə ilə oxuya bilərik — 100 mesajı bir INSERT ilə yazmaq (bölmə 04).
③ Geri təzyiq (backpressure). Növbənin ölçü limiti var — dolarsa, sistem "boğulmaq" əvəzinə xəbərdarlıq verir və biz problemi görürük.

Yəni sap problemini həll edərkən performans və etibarlılıq da qazanırıq.

03 — MQTT modulu

03_src/api/mqtt.py — tam kod

03_src/api/mqtt.py
"""
MQTT qatı — brokerdən mesaj alır, növbəyə qoyur.
paho SİNXRON sapda işləyir; asyncio.Queue ilə async dünyaya körpü qururuq.
"""
import asyncio
import json
import logging

import paho.mqtt.client as mqtt
from pydantic import ValidationError

from .konfiq import MQTT_HOST, MQTT_PORT, TOPIC_BASE
from .modeller import OlcmeIn

log = logging.getLogger("zarat.mqtt")

CLIENT_ID = "zarat-api"
TOPIC_SUB = f"{TOPIC_BASE}/+/olcme"
NOVBE_LIMIT = 5000   # maksimum gözləyən mesaj

# Növbə və event loop — lifespan-da doldurulacaq
novbe: asyncio.Queue | None = None
loop: asyncio.AbstractEventLoop | None = None
client: mqtt.Client | None = None

# MQTT metrikaları
MQTT_METRIKA = {"gelen": 0, "qebul": 0, "redd": 0,
                "novbe_dolu": 0, "qosulub": False}


# ---------- Callback-lər (PAHO SAPINDA işləyir!) ----------
def on_connect(cl, userdata, flags, reason_code, properties):
    if reason_code == 0:
        MQTT_METRIKA["qosulub"] = True
        cl.subscribe(TOPIC_SUB, qos=1)   # abunə MÜTLƏQ burada!
        log.info("MQTT qoşuldu, abunə: %s", TOPIC_SUB)
    else:
        MQTT_METRIKA["qosulub"] = False
        log.error("MQTT qoşulma xətası: %s", reason_code)


def on_disconnect(cl, userdata, flags, reason_code, properties):
    MQTT_METRIKA["qosulub"] = False
    if reason_code != 0:
        log.warning("MQTT kəsildi (%s) — yenidən qoşulur...", reason_code)


def on_message(cl, userdata, msg):
    """
    DİQQƏT: bu funksiya PAHO SAPINDA işləyir, event loop-da YOX.
    Ona görə burada 'await' yazmaq OLMAZ.
    Yeganə işimiz: validasiya + növbəyə qoymaq.
    """
    MQTT_METRIKA["gelen"] += 1

    try:
        xam = json.loads(msg.payload.decode("utf-8"))
        # ★ EYNİ Pydantic modeli — HTTP ilə tam eyni qaydalar ★
        olcme = OlcmeIn.model_validate(xam)

    except (json.JSONDecodeError, UnicodeDecodeError) as e:
        MQTT_METRIKA["redd"] += 1
        log.warning("Pozuq JSON [%s]: %s", msg.topic, e)
        return

    except ValidationError as e:
        MQTT_METRIKA["redd"] += 1
        # Pydantic DƏQİQ deyir: hansı sahə, niyə
        for xeta in e.errors():
            log.warning("Validasiya xətası [%s] %s: %s",
                        msg.topic, xeta["loc"], xeta["msg"])
        return

    # ---- Sap sərhədini keç: paho sapı → event loop ----
    if loop is None or novbe is None:
        return   # servis hələ tam qalxmayıb

    try:
        # call_soon_threadsafe — BAŞQA SAPDAN event loop-a
        # təhlükəsiz müraciət etməyin YEGANƏ düzgün yolu
        loop.call_soon_threadsafe(novbe.put_nowait, olcme)
        MQTT_METRIKA["qebul"] += 1

    except asyncio.QueueFull:
        MQTT_METRIKA["novbe_dolu"] += 1
        log.error("NÖVBƏ DOLDU (%d) — mesaj atıldı! Baza yavaşdır?",
                  NOVBE_LIMIT)


# ---------- Başlatma / dayandırma ----------
def basla(event_loop: asyncio.AbstractEventLoop) -> asyncio.Queue:
    """lifespan-dan çağırılır. Növbəni yaradır, brokerə qoşulur."""
    global novbe, loop, client

    loop  = event_loop
    novbe = asyncio.Queue(maxsize=NOVBE_LIMIT)

    client = mqtt.Client(mqtt.CallbackAPIVersion.VERSION2,
                         client_id=CLIENT_ID,
                         clean_session=False)   # offline növbə!
    client.on_connect    = on_connect
    client.on_disconnect = on_disconnect
    client.on_message    = on_message
    client.reconnect_delay_set(min_delay=1, max_delay=30)

    # API-nin öz LWT-si: sönsə, bunu broker elan edəcək
    client.will_set(f"{TOPIC_BASE}/api/status",
                    payload="offline", qos=1, retain=True)

    client.connect(MQTT_HOST, MQTT_PORT, keepalive=60)
    client.loop_start()   # fon sapı

    client.publish(f"{TOPIC_BASE}/api/status",
                   "online", qos=1, retain=True)

    log.info("MQTT başladı: %s:%s", MQTT_HOST, MQTT_PORT)
    return novbe


def dayandir():
    """lifespan sönəndə çağırılır."""
    if client is None:
        return
    client.publish(f"{TOPIC_BASE}/api/status",
                   "offline", qos=1, retain=True)
    client.loop_stop()
    client.disconnect()
    log.info("MQTT dayandı.")

Bu, bütün modulun ürəyidir.
paho sapından asyncio event loop-una müraciət etməyin yeganə təhlükəsiz yolu budur. Birbaşa novbe.put_nowait(olcme) yazsanız — asyncio.Queue sap-təhlükəsiz olmadığı üçün bəzən işləyər, bəzən data korlanar. Ən pis növ xəta: təsadüfi və təkrarlanmayan.
call_soon_threadsafe isə əməliyyatı event loop-un öz növbəsinə qoyur — loop onu öz sapında icra edir. Təhlükəsiz.
OlcmeIn.model_validate(xam)
Bu, dərsin əsas qazancıdır. HTTP endpoint-i də, MQTT də eyni modeldən keçir. Zonasız vaxt, mətn qiymət, naməlum sahə — hər ikisində eyni cür rədd olunur.
Validasiya qaydasını dəyişmək lazımdırsa, modeller.py-da bir yerdə dəyişirsiniz və hər iki qapıya tətbiq olunur.
maxsize=NOVBE_LIMIT (5000)
Nə: növbənin limiti.
Niyə limit lazımdır: limitsiz növbə — gizli bomba. Baza sönsə, mesajlar yığılar və yaddaş dolana qədər davam edər; sonra proses çökər. Limitlə isə: novbe_dolu sayğacı artır, log xəbərdarlıq verir və biz problemi görürük.
5000 nə deməkdir: 5 cihaz × saniyədə 1 mesaj tempində ≈ 16 dəqiqəlik bufer.
will_set(".../api/status")
API-nin öz LWT-si. Servis çöksə, broker "offline" elan edir. Dashboard bunu görüb "telemetriya servisi işləmir" xəbərdarlığı göstərə bilər.

04 — Performans

Yazıcı: batch INSERT

Növbəyə mesaj düşdü — indi onu bazaya yazan async tapşırıq lazımdır. Amma bir-bir yazmaq israfdır.

Yanaşma100 ölçmə üçünNəticə
Bir-bir INSERT + COMMIT100 tranzaksiya, 100 disk yazmasıYavaş — hər commit disk sinxronizasiyası tələb edir
Batch: 100 sətir, 1 tranzaksiya1 tranzaksiya, 1 disk yazması10–20 dəfə sürətli
03_src/api/yazici.py
"""
Yazıcı — növbədən mesajları dəstə ilə oxuyub bazaya yazır.
FastAPI-nin fon tapşırığı kimi işləyir.
"""
import asyncio
import logging

import psycopg

from .baza import hovuz, HEDLER

log = logging.getLogger("zarat.yazici")

BATCH_OLCU  = 100    # maksimum dəstə ölçüsü
BATCH_VAXT  = 2.0    # saniyə — dəstə dolmasa da bu qədər gözlə

YAZICI_METRIKA = {"yazilan": 0, "anomal": 0,
                  "dublikat": 0, "xeta": 0, "batch": 0}

SQL_OLCME = """
    INSERT INTO olcme (cihaz_kod, olcme_vaxti, qiymet, keyfiyyet)
    VALUES (%s, %s, %s, %s)
    ON CONFLICT (cihaz_kod, olcme_vaxti) DO NOTHING
"""
SQL_XEBER = """
    INSERT INTO xeberdarliq (cihaz_kod, olcme_vaxti, qiymet, novu, mesaj)
    VALUES (%s, %s, %s, %s, %s)
"""


async def deste_topla(novbe: asyncio.Queue) -> list:
    """
    Növbədən dəstə toplayır.
    Ya BATCH_OLCU mesaj yığılana, ya da BATCH_VAXT keçənə qədər.
    """
    deste = []

    # İlk mesajı gözlə (limitsiz — boş növbədə asılı qalırıq, bu normaldır)
    ilk = await novbe.get()
    deste.append(ilk)

    # Qalanlarını vaxt limiti daxilində yığ
    son_vaxt = asyncio.get_event_loop().time() + BATCH_VAXT
    while len(deste) < BATCH_OLCU:
        qalan = son_vaxt - asyncio.get_event_loop().time()
        if qalan <= 0:
            break
        try:
            m = await asyncio.wait_for(novbe.get(), timeout=qalan)
            deste.append(m)
        except asyncio.TimeoutError:
            break   # vaxt bitdi — nə yığılıbsa, onu yaz

    return deste


async def deste_yaz(deste: list) -> None:
    """Dəstəni BİR tranzaksiyada bazaya yazır."""
    olcme_setirleri = []
    xeber_setirleri = []

    for o in deste:
        if o.cihaz_kod not in HEDLER:
            log.warning("Naməlum cihaz: %s — atıldı", o.cihaz_kod)
            YAZICI_METRIKA["xeta"] += 1
            continue

        mn, mx, vahid = HEDLER[o.cihaz_kod]
        keyfiyyet = 1 if mn <= o.qiymet <= mx else 0

        olcme_setirleri.append(
            (o.cihaz_kod, o.olcme_vaxti, o.qiymet, keyfiyyet)
        )

        if o.qiymet > mx:
            xeber_setirleri.append((
                o.cihaz_kod, o.olcme_vaxti, o.qiymet, "yuxari_hedd",
                f"{o.cihaz_kod}: {o.qiymet} {vahid} > maks {mx}",
            ))
            YAZICI_METRIKA["anomal"] += 1
        elif o.qiymet < mn:
            xeber_setirleri.append((
                o.cihaz_kod, o.olcme_vaxti, o.qiymet, "asagi_hedd",
                f"{o.cihaz_kod}: {o.qiymet} {vahid} < min {mn}",
            ))
            YAZICI_METRIKA["anomal"] += 1

    if not olcme_setirleri:
        return

    try:
        async with hovuz.connection() as conn:
            async with conn.cursor() as cur:
                # executemany — bütün dəstə bir gedişdə
                await cur.executemany(SQL_OLCME, olcme_setirleri)
                yazildi = cur.rowcount

                if xeber_setirleri:
                    await cur.executemany(SQL_XEBER, xeber_setirleri)
            # blok bitir → avtomatik COMMIT (bir dəfə!)

        # ON CONFLICT səbəbindən yazılmayanlar = dublikat
        dublikat = len(olcme_setirleri) - max(yazildi, 0)
        if dublikat > 0:
            YAZICI_METRIKA["dublikat"] += dublikat

        YAZICI_METRIKA["yazilan"] += len(olcme_setirleri)
        YAZICI_METRIKA["batch"] += 1

        log.info("Dəstə yazıldı: %d ölçmə, %d alert, %d dublikat",
                 len(olcme_setirleri), len(xeber_setirleri), dublikat)

    except psycopg.Error as e:
        # Tranzaksiya avtomatik ROLLBACK olundu — bütün dəstə itdi.
        # Bu, şüurlu seçimdir: yarımçıq dəstə yazmaqdansa, heç yazma.
        YAZICI_METRIKA["xeta"] += len(olcme_setirleri)
        log.error("Baza xətası, dəstə (%d) itdi: %s",
                  len(olcme_setirleri), e)


async def yazici_dovru(novbe: asyncio.Queue):
    """
    Sonsuz dövr — fon tapşırığı kimi işləyir.
    ÖLMƏMƏLİDİR: hər xətanı tutur və davam edir.
    """
    log.info("Yazıcı başladı (batch=%d, vaxt=%.1fs)",
             BATCH_OLCU, BATCH_VAXT)
    while True:
        try:
            deste = await deste_topla(novbe)
            await deste_yaz(deste)

        except asyncio.CancelledError:
            log.info("Yazıcı dayandırılır...")
            raise   # bu istisnanı UDMA — səliqəli sönmə üçün lazımdır

        except Exception as e:
            # Gözlənilməz xəta — logla, 1 san. gözlə, DAVAM ET
            log.exception("Yazıcıda gözlənilməz xəta: %s", e)
            await asyncio.sleep(1)
deste_topla — iki şərt
Nə: ya 100 mesaj yığılır, ya 2 saniyə keçir — hansı əvvəl olsa.
Niyə iki şərt: yalnız ölçüyə baxsaq, sakit vaxtlarda (dəqiqədə 5 mesaj) dəstə heç vaxt dolmaz və data bazaya saatlarla düşməz. Vaxt limiti bunun qarşısını alır: sakit vaxtda hər 2 saniyədən bir nə varsa yazılır.
Nəticə: yüksək tempdə sürət, aşağı tempdə təzəlik. Hər iki dünyanın yaxşısı.
await novbe.get() — ilk mesaj
Növbə boşdursa, burada asılı qalırıq — və bu, düzgündür. CPU yeyən "boş dövr" yaranmır; event loop bu vaxt başqa işlərlə (HTTP sorğuları) məşğul olur.
executemany(...)
psycopg v3-də bu, sətirləri bir şəbəkə gedişində göndərir. Bir-bir execute() çağırsaydıq, hər biri ayrı gediş-gəliş olardı.
except CancelledError: raise
Kritik incəlik. Servis sönəndə asyncio bu tapşırığa CancelledError göndərir. Onu udsanız (except Exception içində), tapşırıq ölməz və servis sonsuza qədər asılı qalar. Ona görə onu ayrıca tutub raise edirik.
except Exception: sleep(1); davam
Yazıcı 7/24 işləməlidir. Gözlənilməz xəta (məsələn baza tamamilə sönüb) onu öldürməməlidir — logla, bir az gözlə, yenidən cəhd et.

05 — Idempotentlik

Dublikatları öldürmək

Dərs 04-A-da QoS 1 seçəndə demişdik: "mesaj itmir, amma təkrarlana bilər — dublikat problemini bazada həll edəcəyik." İndi o vədin vaxtıdır.

Dublikat haradan gəlir? QoS 1-də publisher mesajı göndərir və broker-dən PUBACK gözləyir. Əgər PUBACK yolda itsə, publisher mesajın çatmadığını düşünüb yenidən göndərir — halbuki broker onu artıq qəbul edib.

Nəticə: eyni ölçmə bazaya iki dəfə düşür. Qrafiklərdə "çift nöqtə", orta hesablamalarda təhrif, ölçmə sayında şişmə.

Həll: təbii unikal açar + ON CONFLICT

Bir cihaz eyni eyni anda iki fərqli dəyər ölçə bilməz. Yəni (cihaz_kod, olcme_vaxti) cütü təbii unikal açardır. Bunu bazaya elan edirik.

02_db/edge/04_unikal.sql — YENİ FAYL
-- ============================================================
-- Idempotentlik: eyni cihaz + eyni vaxt = eyni ölçmə
-- Bu, QoS 1 dublikatlarını bazada həll edir.
-- ============================================================

-- DİQQƏT: partisiyalı cədvəldə UNIQUE məhdudiyyət
-- partisiya açarını (olcme_vaxti) MÜTLƏQ əhatə etməlidir.
-- Bizdə o, onsuz da açarın içindədir — problem yoxdur.

ALTER TABLE olcme
    ADD CONSTRAINT olcme_cihaz_vaxt_unikal
    UNIQUE (cihaz_kod, olcme_vaxti);

-- Yoxlayaq
\d olcme
Əvvəlcə mövcud dublikatları təmizləyin! UNIQUE əlavə etmək uğursuz olacaq, əgər cədvəldə artıq dublikat varsa. Əvvəlcə yoxlayın və təmizləyin:
Dublikatları tapıb təmizləmək
-- 1) Dublikat varmı?
SELECT cihaz_kod, olcme_vaxti, count(*)
FROM olcme
GROUP BY cihaz_kod, olcme_vaxti
HAVING count(*) > 1
LIMIT 10;

-- 2) Varsa — hər cütdən yalnız BİRİNCİSİNİ saxla
DELETE FROM olcme a
USING olcme b
WHERE a.cihaz_kod   = b.cihaz_kod
  AND a.olcme_vaxti = b.olcme_vaxti
  AND a.id > b.id;          -- böyük id-li = sonra gələn = dublikat

-- 3) İndi UNIQUE əlavə et (yuxarıdakı ALTER TABLE)
Yazıcıdakı sətir — indi mənası aydındır
INSERT INTO olcme (cihaz_kod, olcme_vaxti, qiymet, keyfiyyet)
VALUES (%s, %s, %s, %s)
ON CONFLICT (cihaz_kod, olcme_vaxti) DO NOTHING
--   ↑ eyni cüt varsa: xəta vermə, sadəcə YAZMA
Nəticə Eyni mesaj on dəfə gəlsə də, bazada bir sətir olacaq. Sistem idempotent oldu — eyni əməliyyatı təkrarlamaq nəticəni dəyişmir.

Bu, QoS 2-nin bahalı dörd mərhələli əl sıxmasını lazımsız edir — məhz Dərs 04-A-da dediyimiz kimi. Ucuz həll, tam nəticə.

06 — Birləşdirmə

main.py-ın yenilənməsi

İndi lifespan-a MQTT və yazıcı tapşırığını əlavə edirik.

main.py — yenilənmiş lifespan və metrikalar
import asyncio
import logging
from contextlib import asynccontextmanager

from fastapi import FastAPI

from .konfiq import VERSIYA
from .baza import hovuz, HEDLER, hedleri_yukle
from . import mqtt as mqtt_modul
from .yazici import yazici_dovru, YAZICI_METRIKA

log = logging.getLogger("zarat.api")


@asynccontextmanager
async def lifespan(app: FastAPI):
    # ---------- BAŞLANĞIC ----------
    await hovuz.open()
    await hovuz.wait()
    say = await hedleri_yukle()
    log.info("Baza hazır — %d cihaz", say)

    # MQTT-ni başlat və növbəni al
    loop  = asyncio.get_running_loop()
    novbe = mqtt_modul.basla(loop)

    # Yazıcını fon tapşırığı kimi işə sal
    yazici_task = asyncio.create_task(yazici_dovru(novbe))
    log.info("Servis hazır — versiya %s", VERSIYA)

    yield   # ← SERVİS İŞLƏYİR

    # ---------- SÖNMƏ (səliqəli, sıra ilə) ----------
    log.info("Sönmə başladı...")

    # 1) Əvvəlcə MQTT-ni dayandır — yeni mesaj gəlməsin
    mqtt_modul.dayandir()

    # 2) Növbədə qalanları yazmağa imkan ver (maks 5 san.)
    try:
        await asyncio.wait_for(novbe.join(), timeout=5.0)
    except asyncio.TimeoutError:
        log.warning("Növbədə %d mesaj qaldı", novbe.qsize())

    # 3) İndi yazıcını dayandır
    yazici_task.cancel()
    try:
        await yazici_task
    except asyncio.CancelledError:
        pass

    # 4) Sonda hovuzu bağla
    await hovuz.close()
    log.info("Servis söndü.")


app = FastAPI(
    title="Zarat Zavod Telemetriya API",
    version=VERSIYA,
    lifespan=lifespan,
)


# ---------- Metrikalar: indi MQTT-ni də göstərir ----------
@app.get("/metrikalar", tags=["Sistem"])
async def metrikalar():
    novbe = mqtt_modul.novbe
    return {
        "mqtt": {
            **mqtt_modul.MQTT_METRIKA,
            "novbe_uzunlugu": novbe.qsize() if novbe else 0,
        },
        "yazici": YAZICI_METRIKA,
        "cihaz_sayi": len(HEDLER),
        "versiya": VERSIYA,
    }
Sönmə sırasına diqqət — dördü də vacibdir ① MQTT əvvəl dayanır → yeni mesaj gəlmir (növbə artmır).
② Növbədəki qalıq yazılır → data itmir.
③ Yazıcı dayanır → işi bitib.
④ Hovuz bağlanır → bağlantılar səliqəli qaytarılır.

Sıra pozulsa: hovuzu əvvəl bağlasanız, yazıcı növbədəki mesajları yaza bilməz və onlar itər. Bu, "graceful shutdown"dır və istehsalat sistemlərində vacib detaldır.
Kiçik düzəliş: novbe.join() işləməsi üçün novbe.join() yalnız o halda düzgün işləyir ki, yazıcı hər mesaj üçün novbe.task_done() çağırsın. Bunu deste_yaz-dan sonra əlavə edin:

for _ in deste: novbe.task_done()

Əks halda join() həmişə 5 saniyə gözləyəcək (timeout).

03_src/api/loglama.py
"""Loglama konfiqurasiyası — həm ekrana, həm fayla."""
import logging
import logging.handlers
from pathlib import Path

KOK = Path(__file__).resolve().parents[2]
LOG_QOVLUQ = KOK / "_LOG"
LOG_QOVLUQ.mkdir(exist_ok=True)


def qur(seviyye=logging.INFO):
    format = logging.Formatter(
        "%(asctime)s [%(levelname)-7s] %(name)-14s | %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )

    # 1) Ekrana
    ekran = logging.StreamHandler()
    ekran.setFormatter(format)

    # 2) Fayla — 10 MB-da bir fırlanır, 5 köhnə nüsxə saxlanır
    fayl = logging.handlers.RotatingFileHandler(
        LOG_QOVLUQ / "api.log",
        maxBytes=10 * 1024 * 1024,
        backupCount=5,
        encoding="utf-8",
    )
    fayl.setFormatter(format)

    kok = logging.getLogger()
    kok.setLevel(seviyye)
    kok.handlers.clear()
    kok.addHandler(ekran)
    kok.addHandler(fayl)

    # Kitabxanaların səs-küyünü azalt
    logging.getLogger("psycopg.pool").setLevel(logging.WARNING)
