02 — Kök səbəb

Alert hadisə deyil — vəziyyətdir

Bütün problemin kökü bir konseptual səhvdədir. Hazırkı kodumuz belə düşünür:

İndiki məntiq — yazici.py
if o.qiymet > mx:
    xeber_setirleri.append((...))   # HƏR ölçmədə yeni alert

Bu, alerti hadisə (event) kimi görür: "hədd aşıldı → qeyd et". Amma reallıq başqadır.

Hadisə modeli (səhv)Vəziyyət modeli (düzgün)
Sual "Hədd aşıldımı?" "Problem davam edirmi?"
Hər ölçmədə Yeni sətir yaradır Mövcud vəziyyəti yeniləyir
17 saatlıq problem 122 sətir 1 sətir — 17 saat açıq qalıб
Operator görür 122 eyni sətir 1 aktiv alert + "nə vaxtdan bəri", "neçə ölçmə", "pik dəyər"
Problem bitəndə Sadəcə yeni alert gəlmir (bilinmir ki, bitib) Alert bağlanır, bağlanma vaxtı yazılır
Analogiya Xəstəxanada pasiyentin qızdırması var. Həkim hər ölçmədə yeni xəstəlik yazmır — o, bir diaqnoz qoyur və vəziyyəti izləyir: "39,2 → 38,7 → 37,4 → normal". Sonda diaqnozu bağlayır.

Bizim indiki sistemimiz isə hər termometr ölçməsində yeni xəstəlik kartı açır. 122 kart, bir xəstə, bir problem.

Bir sətir — amma 122 sətirdən daha çox məlumat daşıyır. Çünki 122 sətir "pik dəyər nə idi?" sualına cavab vermir; onu tapmaq üçün hamısını skan etmək lazımdır.

03 — Model

Alertin həyat dövrü

Vəziyyət modelində alert doğulur, yaşayır və ölür. Bunu dəqiq təsvir edək.

VəziyyətŞərtSistem nə edir
NORMALDəyər hədlər arasındaHeç nə. Ardıcıllıq sayğacı sıfırlanır.
GÖZLƏMƏ1-2 ardıcıl aşımSayğacı artırır, hələ alert yaratmır
AKTİV3-cü ardıcıl aşımBir alert yaradır (hell_olundu = false)
AKTİV (davam)Aşım davam edirMövcud alerti yeniləyir: tətik sayı ↑, pik dəyər, son dəyər
BAĞLANMIŞDəyər bərpa həddinə düşdüAlerti bağlayır: hell_olundu = true, bağlanma vaxtı
Diqqət: "bərpa həddi" ≠ "hədd" Alert 35 °C-də açılır, amma 35-də bağlanmır — 33-ə düşəndə bağlanır. Bu, hysteresisdir və növbəti bölmələrdə izah edəcəyik. Bunsuz sistem yenidən titrəyəcək.

04 — Debouncing

Bir problem, bir alert

Debouncing — "səkməni söndürmək". Termin elektronikadan gəlir: düymə basılanda kontakt bir neçə dəfə "səkir" və sistem onu 5 basış kimi görür. Həlli: ilk siqnaldan sonra qısa müddət yeni siqnalları saymamaq.

Bizdə eyni məntiq: aktiv alert varsa, yenisini yaratma.

Məntiq — psevdokod
if hədd_aşıldı:
    aktiv = aktiv_alert_tap(cihaz_kod, növ)   # hell_olundu = false olan

    if aktiv:
        # VAR → yenilə, YENİ SƏTİR YARATMA
        aktiv.tetik_sayi += 1
        aktiv.son_qiymet = qiymət
        aktiv.pik_qiymet = max(aktiv.pik_qiymet, qiymət)
    else:
        # YOXDUR → bir alert yarat
        yeni_alert(cihaz_kod, növ, qiymət)

Nəticə: 122 → 1

İndi (debouncing yox)Sonra (debouncing var)
S004 — 17 saat aşım122 sətir1 sətir, tetik_sayi = 122
Operator ekranı122 eyni sətir1 aktiv alert
Məlumat"122 dəfə aşıldı""16:48-dən bəri açıq, 122 ölçmə, pik 9.4, son 8.1"
Baza (ayda, 50 cihaz)~2,6 mln sətirOnlarla sətir
Amma "yeniləmək" nə deməkdir — data itmirmi? Xeyr. Diqqət edin: ölçmələr onsuz da olcme cədvəlində qalır — hər biri, tam tarixçə ilə. Anomal dəyərlər keyfiyyet = 0 bayrağı ilə orada durur.

xeberdarliq cədvəli isə tarixçə deyil — o, operatora göstəriləsi problemlərin siyahısıdır. Onun vəzifəsi datanı saxlamaq deyil, diqqəti yönləndirməkdir.

Bu ayrılıq vacibdir: olcme = fakt, xeberdarliq = şərh.

05 — Hysteresis

Titrəməni dayandırmaq

Debouncing bir problemi həll etdi. İkincisi qalır — və o, daha incədir.

Problem: sərhəddə titrəyən dəyər Temperatur həddi 35 °C-dir. Real dəyər sərhəddə "titrəyir":

34.9 → 35.1 → 34.8 → 35.2 → 34.9 → 35.3 ...

Nə baş verir: alert açılır → bağlanır → açılır → bağlanır. Debouncing burada kömək etmir, çünki hər dəfə yeni problemdir (əvvəlkisi bağlanıб).

Nəticə: yenidən alert seli — sadəcə fərqli mexanizmlə.

Həll: iki fərqli hədd

Hysteresis — açılma və bağlanma hədləri fərqli olur.

DəyərHədd (35)Bərpa (33)Nəticə
36.2aşıb—Alert AÇILIR
34.5aşmır33-dən yuxarıHysteresis zolağında — alert açıq qalır
33.8aşmır33-dən yuxarıHələ zolaqda — açıq
32.6aşmır33-dən aşağıAlert BAĞLANIR
Niyə bu işləyir Titrəyən dəyər (34.9 ↔ 35.2) hysteresis zolağının içindədir — 33-ün altına düşmür. Deməli alert bağlanmır və yenidən açılmır. Bir alert, sabit.

Alert yalnız o zaman bağlanır ki, dəyər həqiqətən normal zonaya qayıtsın (33-ün altına). Bu, "problem həqiqətən bitdi" siqnalıdır — sərhəddə oynamaq yox.

Termostatlar məhz belə işləyir: 20 °C-də qızdırıcını yandırır, 22 °C-də söndürür. 21-də sonsuz yanıб-sönmür.

Zolağın enini necə seçək

ZolaqNəticəNə vaxt uyğundur
Çox dar (35 → 34.9)Titrəmə davam edirHeç vaxt
Diapazonun 5 %-iBalanslıDefault seçimimiz
Çox geniş (35 → 20)Alert problem bitəndən sonra da uzun müddət açıq qalırYalnız çox səs-küylü sensorlar üçün

06 — Ardıcıllıq

Təsadüfi sıçrayışa aldanmamaq

Üçüncü problem: bir anlıq sıçrayış.

Real sensorda dəyər bəzən bir ölçmə üçün "atılır" — elektrik maneəsi, kabel titrəməsi, ADC səhvi. Bu, real problem deyil, ölçmə səs-küyüdür. Amma indiki kodumuz bundan alert yaradır.

Nümunə — S001 temperatur (hədd 35)
12:00:00  →  24.1  ✓ normal
12:00:05  →  24.3  ✓ normal
12:00:10  →  41.7  ⚡ SIÇRAYIŞ — maneə? real problem?
12:00:15  →  24.2  ✓ normal
12:00:20  →  24.4  ✓ normal

# İndiki kod: ALERT yaradır (yalan həyəcan)
# Ardıcıllıq şərti ilə: sayğac 1-ə çatıб sıfırlanır — alert YOX

Həll: ardıcıl N ölçmə şərti

Məntiq
ARDICIL_HEDD = 3   # 3 ardıcıl aşım → alert

if hədd_aşıldı:
    sayğac[cihaz] += 1
    if sayğac[cihaz] >= ARDICIL_HEDD:
        alert_aç_və_ya_yenilə()
else:
    sayğac[cihaz] = 0       # normal dəyər → SIFIRLA
ArdıcıllıqNəticəRisk
1 (indiki)Hər sıçrayış alertYalan həyəcan seli
3 (tövsiyə)15 saniyəlik təsdiq (5 san. tempində)Balanslı
1050 saniyə gecikməReal problem gec aşkarlanır
Bu, kompromisdir — şüurlu seçin Ardıcıllıq şərti gecikmə yaradır: 3 ölçmə × 5 saniyə = 15 saniyə. Yəni real problem 15 saniyə gec aşkarlanacaq.

Temperatur üçün bu, əhəmiyyətsizdir — silo 15 saniyəyə partlamaz.
Amma vibrasiya üçün (yatağın dağılması) 15 saniyə çox ola bilər.

Ona görə ardıcıllıq həddi cihaz tipinə görə fərqli olmalıdır — və biz onu bazada, sensor_tipi cədvəlində saxlayacağıq. Kodda sabit yazmaq səhv olardı.

07 — Səviyyələr

Hər alert bərabər deyil

Hazırda bütün alertlər eynidir. Amma 35,2 °C ilə 48 °C eyni şey deyil.

SəviyyəŞərtReaksiyaBildiriş
xeberdarliq Hədd aşılıb Dashboard-da sarı Yoxdur — operator növbəti baxışda görər
kritik Həddi 20 % aşıb Dashboard-da qırmızı, səs Dərhal — e-poçt / Telegram
cihaz_susur Cihaz N dəqiqədir mesaj göndərmir Dashboard-da boz 5 dəqiqədən sonra
Səviyyə dəyişə bilər — və bu, vacibdir Alert xeberdarliq kimi açılıб, sonra dəyər artsa, kritikə yüksəlməlidir. Vəziyyət modelində bu, təbii işləyir: mövcud alerti yeniləyərkən səviyyəni də yeniləyirik.

Hadisə modelində bu mümkün deyildi — hər sətir "donmuş" idi.

Praktik nəticə: temperatur yavaş-yavaş 36 → 38 → 42-yə qalxsa, operator bir alertin ağırlaşdığını görür — 3 ayrı alert yox.

Kritik həddi necə hesablayaq

# Nisbi (faiz) yanaşma — universaldır
kritik_hedd = max_hedd * 1.20     # 35 → 42

# Temperatur:  35 °C  → kritik 42 °C
# Vibrasiya:    7 mm/s → kritik 8.4 mm/s
# Çəki:       500 kg   → kritik 600 kg

Bu, sadə və universaldır. Amma bəzi sensorlar üçün mütləq hədd lazım ola bilər — ona görə onu da bazada saxlayacağıq (bölmə 11).

08 — Sxem

Yeni xeberdarliq cədvəli

Dörd mexanizm (debouncing, hysteresis, ardıcıllıq, səviyyələr) bazada yeni sütunlar tələb edir.

SütunTipNə üçün
hell_olunduBOOLEANVar idi — Dərs 02-də məhz bunun üçün qoyulmuşdu. İndi işə düşür: false = aktiv alert
seviyyeVARCHARxeberdarliq / kritik
acilma_vaxtiTIMESTAMPTZProblem nə vaxt başladı
baglanma_vaxtiTIMESTAMPTZNə vaxt bitdi (NULL = hələ davam edir)
tetik_sayiINTEGERNeçə ölçmə bu alerti təsdiqləyib
ilk_qiymetNUMERICAlert açılanda dəyər nə idi
son_qiymetNUMERICƏn son dəyər
pik_qiymetNUMERICƏn pis dəyər — operatorun ilk soruşacağı sual
pik_qiymet niyə ayrıca sütundur? Onu olcme cədvəlindən hesablamaq da olardı:
SELECT max(qiymet) FROM olcme WHERE cihaz_kod=... AND olcme_vaxti BETWEEN ...

Amma bu, hər dashboard yeniləməsində milyonlarla sətri skan etmək deməkdir. Pik dəyəri alert yenilənərkən bir dəfə hesablayıб saxlamaq — denormalizasiyadır və burada haqlıdır: yazma bir dəfə, oxuma minlərlə dəfə.

Miqrasiya SQL-i

02_db/edge/05_alert_sxem.sql
-- ============================================================
-- Dərs 06: Alert vəziyyət maşını üçün sxem
-- xeberdarliq: hadisə jurnalı → VƏZİYYƏT cədvəli
-- ============================================================

-- ---------- 1. Yeni sütunlar ----------
ALTER TABLE xeberdarliq
    ADD COLUMN IF NOT EXISTS seviyye        VARCHAR(15)  NOT NULL DEFAULT 'xeberdarliq',
    ADD COLUMN IF NOT EXISTS acilma_vaxti   TIMESTAMPTZ,
    ADD COLUMN IF NOT EXISTS baglanma_vaxti TIMESTAMPTZ,
    ADD COLUMN IF NOT EXISTS tetik_sayi     INTEGER      NOT NULL DEFAULT 1,
    ADD COLUMN IF NOT EXISTS ilk_qiymet     NUMERIC(12,4),
    ADD COLUMN IF NOT EXISTS son_qiymet     NUMERIC(12,4),
    ADD COLUMN IF NOT EXISTS pik_qiymet     NUMERIC(12,4);

-- ---------- 2. Səviyyə yalnız iki dəyər ala bilər ----------
ALTER TABLE xeberdarliq
    DROP CONSTRAINT IF EXISTS xeberdarliq_seviyye_chk;

ALTER TABLE xeberdarliq
    ADD CONSTRAINT xeberdarliq_seviyye_chk
    CHECK (seviyye IN ('xeberdarliq', 'kritik'));

-- ---------- 3. Bağlanma vaxtı yalnız həll olunmuş alertdə ----------
ALTER TABLE xeberdarliq
    DROP CONSTRAINT IF EXISTS xeberdarliq_baglanma_chk;

ALTER TABLE xeberdarliq
    ADD CONSTRAINT xeberdarliq_baglanma_chk
    CHECK (
        (hell_olundu = TRUE  AND baglanma_vaxti IS NOT NULL) OR
        (hell_olundu = FALSE AND baglanma_vaxti IS NULL)
    );
3-cü məhdudiyyət niyə vacibdir O, bazanın yalan danışmasına imkan vermir: "həll olunub, amma nə vaxt bağlandığı bilinmir" və ya "hələ açıqdır, amma bağlanma vaxtı var" — hər ikisi məntiqsizdir və baza onları rədd edir.

Bu, "constraint-driven design"dır: səhv vəziyyəti kodda yoxlamaq əvəzinə, bazada mümkünsüz etmək. Kod səhv etsə də, baza icazə verməz.

09 — Qismən indeks

Debouncing-i bazada məcburi etmək

Debouncing-i kodda yazacağıq. Amma kod səhv edə bilər. Bazada məcbur edək.

02_db/edge/05_alert_sxem.sql — davamı
-- ---------- 4. QİSMƏN UNİKAL İNDEKS — dərsin ürəyi ----------
-- Bir cihaz + bir alert növü üçün YALNIZ BİR aktiv alert ola bilər.
-- WHERE şərti onu "qismən" edir: yalnız AKTİV alertlərə tətbiq olunur.
-- Bağlanmış alertlər istənilən sayda ola bilər (tarixçə).

CREATE UNIQUE INDEX IF NOT EXISTS xeberdarliq_aktiv_unikal
    ON xeberdarliq (cihaz_kod, novu)
    WHERE hell_olundu = FALSE;

-- ---------- 5. Sürətli sorğu üçün indekslər ----------
CREATE INDEX IF NOT EXISTS idx_xeber_aktiv
    ON xeberdarliq (acilma_vaxti DESC)
    WHERE hell_olundu = FALSE;

CREATE INDEX IF NOT EXISTS idx_xeber_cihaz
    ON xeberdarliq (cihaz_kod, acilma_vaxti DESC);
Qismən unikal indeks — niyə bu qədər zərifdir Adi UNIQUE (cihaz_kod, novu) yaramazdı: onda S004 üçün tarixdə ümumiyyətlə bir dənə alert ola bilərdi. Halbuki bizə lazımdır: eyni anda bir aktiv, amma tarixçədə istənilən qədər bağlanmış.

WHERE hell_olundu = FALSE məhz bunu edir: indeks yalnız aktiv sətirləri əhatə edir.

Nəticə: kod səhvən ikinci aktiv alert yaratmağa çalışsa, PostgreSQL onu rədd edir. Debouncing artıq "koddakı yaxşı niyyət" deyil — bazanın qanunudur.

Bu, Dərs 05-dəki UNIQUE (cihaz_kod, olcme_vaxti) ilə eyni fəlsəfədir: düzgün davranışı bazada məcburi et.

Bonus: ON CONFLICT ilə birlikdə işləyir

-- Kodda belə yazacağıq (Hissə B):
INSERT INTO xeberdarliq (cihaz_kod, novu, ...)
VALUES (...)
ON CONFLICT (cihaz_kod, novu) WHERE hell_olundu = FALSE
DO UPDATE SET
    tetik_sayi = xeberdarliq.tetik_sayi + 1,
    son_qiymet = EXCLUDED.son_qiymet,
    pik_qiymet = GREATEST(xeberdarliq.pik_qiymet, EXCLUDED.pik_qiymet);

-- Bir SQL: yoxdursa YARAT, varsa YENİLƏ.
-- "Upsert" adlanır və atomikdir — yarış şəraiti (race condition) yoxdur.

10 — Miqrasiya

390 alerti təmizləmək

Yeni sxem hazırdır. İndi mövcud 390 sətri yeni modelə çevirmək lazımdır — və burada bir maneə var.

Maneə: qismən unikal indeks mövcud data ilə toqquşacaq Hazırda S004 üçün 122 aktiv alert var (hamısında hell_olundu = false). Yeni indeks yalnız birinə icazə verir.

Deməli: əvvəlcə datanı təmizləyirik, sonra indeksi yaradırıq. Sıra vacibdir.

Addım 1 — Vəziyyəti öyrənək (check before changing)

SELECT

  count(*) FILTER (WHERE hell_olundu = false)     AS aktiv,
  count(DISTINCT (cihaz_kod, novu))               AS unikal_problem
FROM xeberdarliq;
"

Addım 3 — Konsolidasiya: 390 → 5

Hər (cihaz_kod, novu) qrupu üçün bir alert saxlayırıq — amma qalanlarındakı məlumatı ona köçürürük. Data itmir, birləşir.

02_db/edge/06_alert_miqrasiya.sql
-- ============================================================
-- 390 hadisə → 5 vəziyyət
-- Hər (cihaz_kod, novu) üçün ən KÖHNƏ sətri saxlayır,
-- qalanlarının məlumatını ona toplayır.
-- ============================================================

BEGIN;

-- 1) Hər qrup üçün aqreqat hesabla və ən köhnə sətrə yaz
WITH aqreqat AS (
    SELECT
        cihaz_kod,
        novu,
        min(id)              AS saxlanacaq_id,
        count(*)             AS tetik_sayi,
        min(olcme_vaxti)     AS acilma_vaxti,
        max(qiymet)          AS pik_qiymet,
        min(yaradilma)       AS ilk_yaradilma
    FROM xeberdarliq
    GROUP BY cihaz_kod, novu
),
son_qiymetler AS (
    -- Hər qrupun ƏN SON qiymətini tap
    SELECT DISTINCT ON (cihaz_kod, novu)
        cihaz_kod, novu, qiymet AS son_qiymet
    FROM xeberdarliq
    ORDER BY cihaz_kod, novu, olcme_vaxti DESC
)
UPDATE xeberdarliq x
SET
    tetik_sayi   = a.tetik_sayi,
    acilma_vaxti = a.acilma_vaxti,
    ilk_qiymet   = x.qiymet,
    pik_qiymet   = a.pik_qiymet,
    son_qiymet   = s.son_qiymet,
    seviyye      = 'xeberdarliq'
FROM aqreqat a
JOIN son_qiymetler s
  ON s.cihaz_kod = a.cihaz_kod AND s.novu = a.novu
WHERE x.id = a.saxlanacaq_id;

-- 2) Qalan dublikatları sil
DELETE FROM xeberdarliq
WHERE id NOT IN (
    SELECT min(id) FROM xeberdarliq GROUP BY cihaz_kod, novu
);

-- 3) Nəticəni gör
SELECT cihaz_kod, novu, seviyye, tetik_sayi,
       acilma_vaxti, ilk_qiymet, pik_qiymet, son_qiymet, hell_olundu
FROM xeberdarliq
ORDER BY tetik_sayi DESC;

11 — Konfiqurasiya

Hədləri hara yazaq?

Dörd mexanizmin hər birinin parametri var: ardıcıllıq həddi, hysteresis zolağı, kritik faiz. Onları kodda sabit yazmaq olmaz — hər sensor tipi üçün fərqli olmalıdır.

Niyə kodda yox? Vibrasiya üçün ardıcıllıq həddi 2 olmalıdır (yataq tez dağılır), temperatur üçün 5 (silo yavaş qızır). Kodda sabit yazsanız, hər dəyişiklik üçün servisi yenidən yazıб deploy etmək lazım gələcək.

Bazada saxlasanız — operator UPDATE edir və /cihazlar/yenile çağırır. Servis restart olmadan.
02_db/edge/05_alert_sxem.sql — davamı
-- ---------- 6. sensor_tipi-yə alert parametrləri ----------

ALTER TABLE sensor_tipi
    ADD COLUMN IF NOT EXISTS ardicil_hedd   SMALLINT      NOT NULL DEFAULT 3,
    ADD COLUMN IF NOT EXISTS hysteresis_faiz NUMERIC(5,2) NOT NULL DEFAULT 5.0,
    ADD COLUMN IF NOT EXISTS kritik_faiz    NUMERIC(5,2)  NOT NULL DEFAULT 20.0;

COMMENT ON COLUMN sensor_tipi.ardicil_hedd IS
    'Neçə ardıcıl aşımdan sonra alert yaransın';
COMMENT ON COLUMN sensor_tipi.hysteresis_faiz IS
    'Bərpa həddi: diapazonun neçə faizi geri çəkilsin';
COMMENT ON COLUMN sensor_tipi.kritik_faiz IS
    'Həddi neçə faiz aşanda "kritik" sayılsın';

-- ---------- 7. Tiplərə uyğun dəyərlər ----------

-- Vibrasiya: TEZ reaksiya (yataq dağıla bilər)
UPDATE sensor_tipi SET ardicil_hedd = 2, kritik_faiz = 15.0
WHERE kod = 'VIBRASIYA';

-- Temperatur: yavaş dəyişir, səbirli olaq
UPDATE sensor_tipi SET ardicil_hedd = 5, hysteresis_faiz = 6.0
WHERE kod = 'TEMP';

-- Çəki: səs-küylü sensor, geniş hysteresis
UPDATE sensor_tipi SET ardicil_hedd = 3, hysteresis_faiz = 8.0
WHERE kod = 'CEKI';

-- Enerji: qəfil sıçrayışlar normaldır (mühərrik işə düşür)
UPDATE sensor_tipi SET ardicil_hedd = 4
WHERE kod = 'ENERJI';

-- Yoxla
SELECT kod, ad, min_hedd, max_hedd, vahid,
       ardicil_hedd, hysteresis_faiz, kritik_faiz
FROM sensor_tipi ORDER BY kod;

Hesablama düsturları

# Kodda (Hissə B) belə hesablanacaq:

diapazon    = max_hedd - min_hedd

# Bərpa həddi (hysteresis)
berpa_yuxari = max_hedd - diapazon * (hysteresis_faiz / 100)
berpa_asagi  = min_hedd + diapazon * (hysteresis_faiz / 100)

# Kritik hədd
kritik_yuxari = max_hedd * (1 + kritik_faiz / 100)
kritik_asagi  = min_hedd * (1 - kritik_faiz / 100)

# TEMP nümunəsi: min=-5, max=35, hysteresis=6%, kritik=20%
#   diapazon      = 40
#   berpa_yuxari  = 35 - 40*0.06  = 32.6   ← alert 32.6-da bağlanır
#   kritik_yuxari = 35 * 1.20     = 42.0   ← 42-dən yuxarı = KRİTİK
