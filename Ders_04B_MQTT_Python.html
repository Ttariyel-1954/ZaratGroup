Client-in dörd mərhələsi

Hər MQTT client eyni həyat dövrünü yaşayır. Bunu başa düşsəniz, həm publisher, həm subscriber kodu aydın olacaq.

MərhələMetodNə edir
1. Yaratmamqtt.Client(...)Client obyektini qurur — ID, sessiya rejimi, protokol versiyası
2. Hazırlıqwill_set(...)
on_connect = ...		LWT elan olunur, callback funksiyaları bağlanır
3. Qoşulmaconnect()
loop_start()		Brokerə qoşulur və fon sapını (thread) başladır
4. İşpublish() / on_messageMesaj göndərir və ya qəbul edir
loop_start() vs loop_forever() — ən çox səhv edilən yer MQTT client-in daim işləyən bir dövrü olmalıdır: o, şəbəkəni dinləyir, keepalive nəbzini göndərir, yenidən qoşulur. Bunsuz client "ölür".

loop_start() — dövrü ayrı sapda (thread) başladır, əsas proqram davam edir. Publisher üçün: biz öz döngəmizdə ölçmə yaradıб göndəririk.
loop_forever() — dövrü əsas sapda işlədir və proqramı bloklayır. Subscriber üçün: körpünün başqa işi yoxdur — sadəcə gözləyib mesaj alır.

Bu ikisini qarışdırmaq ən çox rast gəlinən səhvdir: loop_start() yazıб sonra proqramı bitirsəniz, mesajlar göndərilmədən proses ölür.
Versiya tələsi — paho-mqtt 2.x paho-mqtt 2.0-da callback imzaları dəyişdi. İnternetdəki köhnə nümunələr (2024-dən əvvəl) işləməyəcək və TypeError verəcək.

Həll: client yaradanda versiyanı açıq göstərin —
mqtt.Client(mqtt.CallbackAPIVersion.VERSION2, ...)
Aşağıdakı bütün kod məhz bu API ilə yazılıb.

03 — Publisher

sensor_simulyator.py — tam kod

Bu proqram real gateway-in rolunu oynayır: cihazları bazadan oxuyur (real cihaz siyahısı ilə işləmək üçün), hər biri üçün məntiqli dəyər yaradır və brokerə göndərir.

03_src/publisher/sensor_simulyator.py
#!/usr/bin/env python3
"""
Zarat Faza 2 — Sensor simulyatoru (PUBLISHER)
Rolu: real zavodda bu, gateway/PLC olacaq.
Cihazları bazadan oxuyur, ölçmə yaradır, MQTT brokerə göndərir.
Bazaya YAZMIR — yalnız oxuyur (cihaz siyahısı üçün).
"""
import os, sys, json, time, random, signal
from datetime import datetime, timezone
from pathlib import Path

import psycopg
from dotenv import load_dotenv
import paho.mqtt.client as mqtt

# ---- 1. Konfiqurasiya (.env-dən) ----
KOK = Path(__file__).resolve().parents[2]
load_dotenv(KOK / "01_config" / ".env")

MQTT_HOST  = os.getenv("MQTT_HOST", "localhost")
MQTT_PORT  = int(os.getenv("MQTT_PORT", "1883"))
TOPIC_BASE = os.getenv("MQTT_TOPIC_BASE", "zavod/siyezen")
INTERVAL   = float(os.getenv("SIM_INTERVAL", "5"))   # saniyə

EDGE_DSN = (
    f"host={os.getenv('EDGE_DB_HOST')} "
    f"port={os.getenv('EDGE_DB_PORT')} "
    f"dbname={os.getenv('EDGE_DB_NAME')} "
    f"user={os.getenv('EDGE_DB_USER')}"
)

CLIENT_ID = "zarat-simulyator"
isleyir = True   # Ctrl+C üçün bayraq


# ---- 2. Cihazları bazadan oxu ----
def cihazlari_oxu():
    """Aktiv cihazları və onların hədlərini bazadan gətirir."""
    sql = """
        SELECT c.kod, t.kod AS tip, t.min_hedd, t.max_hedd, t.vahid
        FROM cihaz c
        JOIN sensor_tipi t ON c.sensor_tipi_kod = t.kod
        WHERE c.status = 'aktiv'
        ORDER BY c.kod
    """
    with psycopg.connect(EDGE_DSN) as conn, conn.cursor() as cur:
        cur.execute(sql)
        setirler = cur.fetchall()

    cihazlar = []
    for kod, tip, mn, mx, vahid in setirler:
        mn, mx = float(mn), float(mx)
        cihazlar.append({
            "kod":   kod,
            "tip":   tip,
            "min":   mn,
            "max":   mx,
            "vahid": vahid,
            "cari":  (mn + mx) / 2,        # başlanğıc: orta nöqtə
            "addim": (mx - mn) * 0.02,     # bir addımın maks. ölçüsü
        })
    return cihazlar


# ---- 3. Növbəti dəyər (random walk) ----
def novbeti_qiymet(c):
    """Əvvəlki dəyərdən kiçik addımla sürüşür — real sensor kimi."""
    delta = random.uniform(-c["addim"], c["addim"])
    yeni  = c["cari"] + delta

    # 2% ehtimalla anomaliya sıçrayışı → Dərs 6-nı sınamaq üçün
    if random.random() < 0.02:
        yeni = c["max"] * random.uniform(1.05, 1.20)

    # həddən çox uzaqlaşmasın (fizika məntiqi)
    genislik = c["max"] - c["min"]
    asagi  = c["min"] - genislik * 0.3
    yuxari = c["max"] + genislik * 0.3
    c["cari"] = max(asagi, min(yuxari, yeni))

    return round(c["cari"], 3)


# ---- 4. MQTT callback-ləri (paho 2.x imzaları!) ----
def on_connect(client, userdata, flags, reason_code, properties):
    if reason_code == 0:
        print(f"[MQTT] Qoşuldu: {MQTT_HOST}:{MQTT_PORT}")
        # Qoşulan kimi "online" statusunu retain ilə elan et
        client.publish(f"{TOPIC_BASE}/simulyator/status",
                       "online", qos=1, retain=True)
    else:
        print(f"[MQTT] XƏTA — qoşulma alınmadı: {reason_code}")


def on_disconnect(client, userdata, flags, reason_code, properties):
    if reason_code != 0:
        print(f"[MQTT] Gözlənilməz kəsilmə ({reason_code}) — yenidən qoşulur...")


# ---- 5. Ctrl+C-ni səliqəli tut ----
def dayandir(signum, frame):
    global isleyir
    print("\n[SİM] Dayandırılır...")
    isleyir = False

signal.signal(signal.SIGINT, dayandir)


# ---- 6. Əsas döngə ----
def main():
    cihazlar = cihazlari_oxu()
    if not cihazlar:
        print("[XƏTA] Bazada aktiv cihaz yoxdur. 03_seed.sql işlədilibmi?")
        sys.exit(1)
    print(f"[SİM] {len(cihazlar)} aktiv cihaz tapıldı: "
          + ", ".join(c["kod"] for c in cihazlar))

    # Client yarat — VERSION2 API, sabit ID
    client = mqtt.Client(mqtt.CallbackAPIVersion.VERSION2,
                         client_id=CLIENT_ID)
    client.on_connect    = on_connect
    client.on_disconnect = on_disconnect

    # LWT — "vəsiyyət": mən yox olsam, bunu sən göndər
    client.will_set(f"{TOPIC_BASE}/simulyator/status",
                    payload="offline", qos=1, retain=True)

    # Avtomatik yenidən qoşulma: 1 san-dən 30 san-ə qədər artan gözləmə
    client.reconnect_delay_set(min_delay=1, max_delay=30)

    client.connect(MQTT_HOST, MQTT_PORT, keepalive=30)
    client.loop_start()   # fon sapı — şəbəkəni idarə edir

    sayqac = 0
    while isleyir:
        indi = datetime.now(timezone.utc).isoformat()

        for c in cihazlar:
            qiymet = novbeti_qiymet(c)
            topic  = f"{TOPIC_BASE}/{c['kod']}/olcme"
            payload = json.dumps({
                "cihaz_kod":   c["kod"],
                "qiymet":      qiymet,
                "olcme_vaxti": indi,
                "vahid":       c["vahid"],
            })

            netice = client.publish(topic, payload, qos=1, retain=True)
            if netice.rc != mqtt.MQTT_ERR_SUCCESS:
                print(f"[XƏTA] Göndərilmədi: {topic} (rc={netice.rc})")
            else:
                sayqac += 1
                print(f"  → {c['kod']:6} {qiymet:9.3f} {c['vahid']}")

        print(f"[SİM] Cəmi göndərilib: {sayqac}\n")
        time.sleep(INTERVAL)

    # Səliqəli çıxış: "offline" elan et, sonra ayrıl
    client.publish(f"{TOPIC_BASE}/simulyator/status",
                   "offline", qos=1, retain=True)
    time.sleep(0.5)          # mesajın çıxmasına imkan ver
    client.loop_stop()
    client.disconnect()
    print(f"[SİM] Bitdi. Ümumi: {sayqac} mesaj.")


if __name__ == "__main__":
    main()

04 — Data keyfiyyəti

Random walk: niyə sadə random.uniform() yaramır?

Ən sadə yanaşma belə olardı:

SƏHV yanaşma
qiymet = random.uniform(mn, mx)   # hər dəfə tam təsadüfi

Bu, işləyər — amma yaratdığı data yalandır və bütün gələcək işi korlayar. Səbəb: real dünyada temperatur bir saniyədə 5°C-dən 34°C-ə tullanmır. O, tədricən sürüşür.

YanaşmaData necə görünürNəticə
random.uniform() Xaotik zil-zil — qrafik "diş" kimi Trend təhlili mənasızdır; anomaliya aşkarlaması işləmir (hər şey anomaliyadır)
Random walk Yumşaq sürüşən əyri — real sensor kimi Trend görünür; anomaliya seçilir; qrafiklər inandırıcıdır
Məntiq yeni = cari + kiçik_təsadüfi_addım

Addımın ölçüsü diapazonun 2 %-idir. Temperatur üçün (−5…35 °C, diapazon 40) bu, ±0,8 °C deməkdir — real siloda 5 saniyədə baş verə biləcək dəyişiklik. Dəyər əvvəlki dəyərdən "yapışır" və təbii axın yaranır.

2 % anomaliya sıçrayışı isə bilərəkdən qoyulub: Dərs 6-da anomaliya sistemini qurduqda onu sınamaq üçün real hədd aşımları lazımdır. Sırf normal data ilə alert kodunun işlədiyini heç vaxt yoxlaya bilməzdik.

Sonra buraya nə əlavə edə bilərik

  • Gündəlik dövr: temperatur gündüz qalxsın, gecə düşsün — math.sin() ilə saat əsaslı əyri.
  • Cihaz nasazlığı: təsadüfən sensorun "ilişməsi" (eyni dəyəri təkrarlaması) — bu, real dünyada olur və aşkarlanması ayrıca bir bacarıqdır.
  • Korrelyasiya: vibrasiya qalxanda enerji sərfiyyəti də qalxsın — real fizikaya uyğun.
  • Növbə rejimi: gecə xətt dayanır → çəki və enerji sıfıra düşür.

05 — Müqavilə

Payload formatı: publisher ilə körpü arasında müqavilə

Broker payload-ın içinə baxmır. Yəni formatın düzgünlüyünə heç kim nəzarət etmir — bu, tamamilə bizim məsuliyyətimizdir. Ona görə formatı sənəd kimi qeyd edirik.

Ölçmə mesajının formatı — DƏYİŞMƏZ MÜQAVİLƏ
Topic:   zavod/siyezen/{cihaz_kod}/olcme
QoS:     1
Retain:  true

Payload (JSON):
{
  "cihaz_kod":   "S001",                        // məcburi — cihaz cədvəlində olmalıdır
  "qiymet":      24.317,                       // məcburi — ədəd (int və ya float)
  "olcme_vaxti": "2026-07-11T09:15:00+00:00",  // məcburi — ISO 8601, zaman zonası ilə
  "vahid":       "°C"                          // opsional — yalnız məlumat üçün
}
Niyə "zaman zonası ilə" bu qədər vurğulanır? olcme cədvəlində sütun tipi TIMESTAMPTZ-dir (zonalı). Əgər zonasız vaxt göndərsək ("2026-07-11T09:15:00"), PostgreSQL onu serverin zonası ilə şərh edir. Zavod və mərkəz fərqli zonalarda olsa (yaxud yay vaxtı dəyişsə), ölçmələr səhv saata düşər və heç kim səbəbini tapa bilməz.

Ona görə kodda datetime.now(timezone.utc).isoformat() istifadə edirik — həmişə UTC, həmişə açıq zona.
Bu müqavilənin real dəyəri Bu format sizin ilə gələcək gateway inteqratoru arasındakı sənəddir (Dərs 04-T-də danışdığımız TZ-nin bir hissəsi). Gateway məhz bu topic-ə, məhz bu sahələrlə yazacaq. Əgər inteqrator "value" yazsa, "qiymet" yox — körpü mesajı oxuya bilməyəcək. Formatı əvvəlcədən yazıб təhvil vermək bu problemi yaranmamış həll edir.

#!/usr/bin/env python3
"""
Zarat Faza 2 — MQTT → PostgreSQL körpüsü (SUBSCRIBER)
Brokerdən ölçmələri alır, validasiyadan keçirir, olcme cədvəlinə yazır.
Hədd aşılanda xeberdarliq yaradır.
Dərs 05-də bu məntiq FastAPI servisinə köçəcək.
"""
import os, json, signal, sys
from datetime import datetime
from pathlib import Path

import psycopg
from dotenv import load_dotenv
import paho.mqtt.client as mqtt

KOK = Path(__file__).resolve().parents[2]
load_dotenv(KOK / "01_config" / ".env")

MQTT_HOST  = os.getenv("MQTT_HOST", "localhost")
MQTT_PORT  = int(os.getenv("MQTT_PORT", "1883"))
TOPIC_BASE = os.getenv("MQTT_TOPIC_BASE", "zavod/siyezen")

EDGE_DSN = (
    f"host={os.getenv('EDGE_DB_HOST')} "
    f"port={os.getenv('EDGE_DB_PORT')} "
    f"dbname={os.getenv('EDGE_DB_NAME')} "
    f"user={os.getenv('EDGE_DB_USER')}"
)

CLIENT_ID = "zarat-kopru"          # SABİT — offline növbə üçün!
TOPIC_SUB = f"{TOPIC_BASE}/+/olcme"  # yalnız ölçmələr, "#" YOX!

SQL_OLCME = """
    INSERT INTO olcme (cihaz_kod, olcme_vaxti, qiymet, keyfiyyet)
    VALUES (%s, %s, %s, %s)
"""
SQL_XEBER = """
    INSERT INTO xeberdarliq (cihaz_kod, olcme_vaxti, qiymet, novu, mesaj)
    VALUES (%s, %s, %s, %s, %s)
"""

conn = None
hedler = {}      # {cihaz_kod: (min, max, vahid)}
sayqac = {"ok": 0, "xeta": 0, "alert": 0}


# ---- Hədləri bir dəfə yaddaşa yüklə ----
def hedleri_yukle():
    """Hər mesajda bazaya sorğu atmamaq üçün hədləri kеşləyirik."""
    sql = """
        SELECT c.kod, t.min_hedd, t.max_hedd, t.vahid
        FROM cihaz c JOIN sensor_tipi t ON c.sensor_tipi_kod = t.kod
    """
    with conn.cursor() as cur:
        cur.execute(sql)
        for kod, mn, mx, vahid in cur.fetchall():
            hedler[kod] = (float(mn), float(mx), vahid)
    print(f"[KÖRPÜ] {len(hedler)} cihazın həddi yükləndi.")


# ---- Callback: qoşuldu ----
def on_connect(client, userdata, flags, reason_code, properties):
    if reason_code == 0:
        print(f"[MQTT] Qoşuldu. Abunə: {TOPIC_SUB}")
        # Abunə MÜTLƏQ burada olmalıdır — on_connect içində!
        client.subscribe(TOPIC_SUB, qos=1)
    else:
        print(f"[MQTT] Qoşulma xətası: {reason_code}")


# ---- Callback: mesaj gəldi — ƏSAS İŞ BURADADIR ----
def on_message(client, userdata, msg):
    global conn
    try:
        # 1) JSON-u aç
        data = json.loads(msg.payload.decode("utf-8"))

        # 2) Məcburi sahələri yoxla
        kod   = data["cihaz_kod"]
        qiy   = float(data["qiymet"])
        vaxt  = datetime.fromisoformat(data["olcme_vaxti"])

        # 3) Cihaz tanınırmı?
        if kod not in hedler:
            print(f"[XƏBƏRDARLIQ] Naməlum cihaz: {kod} — atıldı")
            sayqac["xeta"] += 1
            return

        mn, mx, vahid = hedler[kod]

        # 4) Keyfiyyət bayrağı: hədd aşılıbsa 0, normaldırsa 1
        keyfiyyet = 1 if mn <= qiy <= mx else 0

        # 5) Bazaya yaz (ölçmə + lazımsa xəbərdarlıq — BİR tranzaksiyada)
        with conn.cursor() as cur:
            cur.execute(SQL_OLCME, (kod, vaxt, qiy, keyfiyyet))

            if qiy > mx:
                cur.execute(SQL_XEBER, (kod, vaxt, qiy, "yuxari_hedd",
                    f"{kod}: {qiy} {vahid} > maks {mx} {vahid}"))
                sayqac["alert"] += 1
                print(f"  ⚠ ALERT {kod}: {qiy} {vahid} (maks {mx})")
            elif qiy < mn:
                cur.execute(SQL_XEBER, (kod, vaxt, qiy, "asagi_hedd",
                    f"{kod}: {qiy} {vahid} < min {mn} {vahid}"))
                sayqac["alert"] += 1
                print(f"  ⚠ ALERT {kod}: {qiy} {vahid} (min {mn})")

        conn.commit()      # hər ikisi birlikdə təsdiqlənir
        sayqac["ok"] += 1
        print(f"  ✓ {kod:6} {qiy:9.3f} {vahid}   "
              f"[ok={sayqac['ok']} alert={sayqac['alert']} xeta={sayqac['xeta']}]")

    except (KeyError, ValueError, json.JSONDecodeError) as e:
        # Pozuq mesaj — logla, amma servisi ÖLDÜRMƏ
        print(f"[XƏTA] Pozuq mesaj ({msg.topic}): {e}")
        sayqac["xeta"] += 1

    except psycopg.Error as e:
        # Baza xətası — geri qaytar, bağlantını bərpa etməyə çalış
        print(f"[BAZA XƏTASI] {e}")
        sayqac["xeta"] += 1
        try:
            conn.rollback()
        except Exception:
            print("[BAZA] Bağlantı bərpa edilir...")
            conn = psycopg.connect(EDGE_DSN)


def dayandir(signum, frame):
    print(f"\n[KÖRPÜ] Dayandırılır. Yekun: {sayqac}")
    sys.exit(0)

signal.signal(signal.SIGINT, dayandir)


def main():
    global conn
    conn = psycopg.connect(EDGE_DSN)
    print(f"[BAZA] Qoşuldu: {os.getenv('EDGE_DB_NAME')}")
    hedleri_yukle()

    client = mqtt.Client(mqtt.CallbackAPIVersion.VERSION2,
                         client_id=CLIENT_ID,
                         clean_session=False)   # ← offline növbə üçün!
    client.on_connect = on_connect
    client.on_message = on_message
    client.reconnect_delay_set(min_delay=1, max_delay=30)

    client.connect(MQTT_HOST, MQTT_PORT, keepalive=60)
    client.loop_forever()   # əsas sapda — başqa işimiz yoxdur


if __name__ == "__main__":
    main()

Ən vacib qərarlar

clean_session=False
Nə: brokerə deyir ki, sessiyanı yadda saxla.
Niyə həyati: körpü söndürülsə (yenilənmə, restart, çökmə), broker o müddətdə gələn mesajları onun üçün növbəyə yığır və qayıdanda çatdırır. Bu olmasa, körpünün söndüyü hər saniyənin sensor məlumatı həmişəlik itir. Sabit client_id ilə birlikdə işləyir — biri olmasa, digəri mənasızdır.
client.subscribe() — on_connect içində
Niyə burada, main-də yox: qoşulma kəsilib bərpa olunanda abunə itir. on_connect hər yenidən qoşulmada çağırıldığı üçün abunə avtomatik bərpa olunur. main()-də yazsanız — ilk kəsilmədən sonra körpü qoşulu görünər, amma heç nə almaz. Bu, tapılması ən çətin xətalardandır.
hedleri_yukle() — keş
Niyə: hər mesajda hədləri bazadan soruşsaq, saniyədə onlarla lüzumsuz sorğu olar. Hədlər nadir dəyişir — bir dəfə yükləyib yaddaşda saxlayırıq.
Kompromis: yeni cihaz əlavə edəndə körpünü restart etmək lazımdır. Kiçik qiymət, böyük qazanc.
keyfiyyet = 1 if mn <= qiy <= mx else 0
Nə: Dərs 02-də olcme.keyfiyyet sütununu məhz bunun üçün yaratmışdıq.
Niyə dəyəri atmırıq: anomal dəyər də faktdır — silo həqiqətən 40°C-yə qalxıb. Onu bazadan silmək = həqiqəti gizlətmək. Ona görə yazırıq, amma bayraqlayırıq: dashboard normal qrafiki keyfiyyet=1 ilə çəkir, anomaliyaları isə ayrıca göstərir.
try / except — üç səviyyə
Niyə ayrı-ayrı tuturuq:
• JSONDecodeError/KeyError → pozuq mesaj. Logla, at, davam et. Bir pozuq mesaj bütün servisi öldürməməlidir.
• psycopg.Error → baza problemi. rollback() et; bağlantı ölübsə, yenidən qur.
Ümumi prinsip: körpü 7/24 işləyən servisdir. O, heç bir halda ölməməlidir — yalnız logla və işləməyə davam et.
conn.commit() — sonda, bir dəfə
Nə: ölçmə və xəbərdarlıq bir tranzaksiyada təsdiqlənir.
Niyə: ölçmə yazılıб, xəbərdarlıq yazılmasa — baza yalan danışır (anomal dəyər var, amma alert yoxdur). Ya hər ikisi, ya heç biri.

07 — Alert

Hədd yoxlaması: sadə, amma dolğun

Körpüdəki alert məntiqi hazırda ən sadə formadadır — hər hədd aşımı üçün bir xəbərdarlıq yaradır. Bu, işləyir, amma bir problemi var və onu indi tanımaq vacibdir.

Problem: "alert seli" (alert storm) Temperatur 36 °C-də ilişib qalsa və simulyator hər 5 saniyədə göndərsə — dəqiqədə 12 xəbərdarlıq, saatda 720. Bir günə xeberdarliq cədvəlində 17 min sətir. Operator bu selin içində əsl vacib hadisəni görməyəcək.

Bu, indi problem deyil — sınaq mərhələsindəyik və hər aşımı görmək istəyirik. Amma Dərs 6-da bunu həll edəcəyik.

Dərs 6-da nə əlavə edəcəyik

  • Debouncing: aktiv (həll olunmamış) xəbərdarlıq varsa, yenisini yaratma — mövcudu yenilə. hell_olundu sütunu məhz bunun üçün var.
  • Hysteresis: hədd 35 °C-dirsə, alert 35-də açılsın, amma yalnız 33-ə düşəndə bağlansın. Əks halda 34,9 ↔ 35,1 arasında titrəyən dəyər onlarla alert yaradar.
  • Ardıcıllıq şərti: bir anlıq sıçrayış (elektrik maneəsi) alert yaratmasın — ardıcıl 3 ölçmə həddi aşarsa yaransın.
  • Səviyyələr: xəbərdarlıq (həddə yaxın) / kritik (həddi 20 % aşıb) — reaksiya fərqli olmalıdır.
  • Bildiriş: e-poçt / Telegram / SMS — kritik alert operatora dərhal çatsın.
  • cihaz_susur: LWT status mesajlarını dinləyib susan cihazı aşkarlamaq (Hissə A, bölmə 05).

11 — Gələcək

Buradan hara gedirik?

Bu körpü işləyən, amma sadə bir servisdir. Növbəti dərslərdə hər hissəsi güclənəcək — və indi hara nə əlavə olunacağını dəqiq bilirik.

  • Dərs 05 — FastAPI: körpünün məntiqi HTTP servisinə köçəcək. Nə qazanırıq: sağlamlıq yoxlaması (/health), metrikalar, HTTP ilə əl ilə ölçmə əlavə etmək imkanı, və Pydantic ilə ciddi validasiya (indiki try/except əvəzinə).
  • Dərs 06 — Anomaliya: debouncing, hysteresis, alert səviyyələri, cihaz_susur (LWT status dinləmə), bildiriş göndərmə.
  • Dərs 07 — Sinxronizasiya: sync_status=0 olan sətirləri mərkəzi zarat_erp_2-yə köçürmək (outbox pattern), təkrar cəhd və konflikt idarəsi.
  • Batch INSERT: hazırda hər mesaj ayrıca INSERT-dir. Yüksək tempdə (saniyədə yüzlərlə) bu yavaşdır — mesajları 100-lük dəstələrə yığıб executemany() ilə yazmaq lazım gələcək.
  • Idempotentlik: QoS 1 dublikat yarada bilər. ON CONFLICT (cihaz_kod, olcme_vaxti) DO NOTHING əlavə etmək üçün əvvəlcə həmin cütə UNIQUE məhdudiyyət qoyulmalıdır.
  • Strukturlu loglama: print() əvəzinə logging modulu — səviyyələr, fayl rotasiyası, timestamp.
  • Servis kimi işlətmək: körpü 7/24 işləməlidir — launchd (Mac) və ya systemd (Linux) ilə avtomatik başlatma və çökəndə yenidən qaldırma.
  • Real gateway: simulyator silinir, yerinə fiziki gateway gəlir. Körpü dəyişmir — çünki müqavilə (topic + JSON) eynidir. Dərs 04-T-də danışdığımız təhvil məhz bu andır.
