Giriş və tam memarlıq

Bu, Dərs 1-in tam və genişləndirilmiş versiyasıdır. Sensor məlumatlarının zavod sahəsindən çıxıb mərkəzi ERP-yə çatana qədər keçdiyi bütün yolu — MQTT broker, FastAPI xidməti, lokal və mərkəzi bazalar daxil olmaqla — sıfırdan qururuq. Sonda tam işlək layihə qovluğu yaradılacaq və 10 dərsin hər biri öz yerini tapacaq.

01

Faza 1-dən Faza 2-yə keçid

Faza 1-də Zarat Group-un strukturunu rəqəmsallaşdırdıq: təşkilat, tərəf-müqabillər, məhsullar, maliyyə hesabları — müəssisənin «kim, nə, harada» sualına cavab verən master data. Bu məlumat nisbətən statikdir: bir işçi qəbul olunur, bir anbar açılır.

Faza 2 fərqli təbiətli məlumatla işləyir: zavodun fiziki proseslərindən fasiləsiz axan göstəricilər — temperatur, rütubət, çəki, vibrasiya, enerji. Bu, ERP-nin «canlı» damarıdır. Mənbə insan deyil, maşındır və axın kəsilmir.

Niyə vacibdir
Siyəzən yem fabrikində qarışdırıcının temperaturu həddi aşarsa, bunu dərhal bilmək lazımdır — 5 dəqiqə sonra deyil. Ona görə Faza 2-nin özəyi sənəd yazıb-oxumaq deyil, davamlı hadisə axınının (event stream) etibarlı idarə olunmasıdır.
02

Tam memarlıq — beş mərhələli axın

Sistemi beş aydın mərhələyə bölürük. Hər mərhələ öz məsuliyyətini daşıyır; biri müvəqqəti dayansa belə, digərləri işini davam etdirir. Bu, sənaye IoT sistemlərində «edge-first, loosely coupled» adlanan yanaşmadır — komponentlər bir-birinə sıx bağlı deyil, mesaj vasitəsilə ünsiyyət qurur.

Diaqram: sensor siqnalı əvvəlcə MQTT vasitəsilə yayımlanır, FastAPI onu tutub yoxlayır və zavod bazasına yazır. Anomaliya aşkarlansa xəbərdarlıq yaranır. Ayrı bir sinxronizasiya servisi məlumatı vaxtaşırı mərkəzi bazaya köçürür — şəbəkə kəsilsə belə, zavod öz işinə davam edir.

03

MQTT nədir və niyə seçdik?

MQTT (Message Queuing Telemetry Transport) — az enerji sərf edən və zəif internet bağlantısına malik cihazlar üçün nəzərdə tutulmuş yüngül mesaj protokoludur. Zavod şəraitində sensorlar bir-biri ilə birbaşa danışmır — ortada Broker adlı vasitəçi server dayanır.

AnlayışNə deməkdirZarat üçün nümunə
BrokerBütün mesajları qəbul edib müvafiq abunəçilərə ötürən mərkəzi serverMosquitto — zavod kompüterində işə düşür
TopicMesajların kateqoriya üzrə axdığı kanalzavod/siyezen/temperatur, zavod/siyezen/rutubet
PublishMesaj göndərməkTemperatur sensoru hər 2 saniyədə oxunuşunu yayımlayır
SubscribeMesajı dinləməkFastAPI xidməti zavod/siyezen/# mövzusuna abunə olur
QoSÇatdırılma zəmanəti səviyyəsi (0/1/2)Temperatur → QoS 0 (itsə problem deyil), Xəbərdarlıq → QoS 1-2
Last WillCihaz qəflətən oflayn olarsa, broker avtomatik xəbər verirSensor 14 cavab verməyəndə dərhal bilinir
Niyə birbaşa yazmırıq
Sensor birbaşa PostgreSQL-ə yazsaydı, hər sensor bazanın parolunu bilməli, bağlantı məntiqini idarə etməli olardı. MQTT bu mürəkkəbliyi aradan qaldırır: sensor sadəcə «mən bu ədədi gördüm» deyir, qalanını broker və FastAPI həll edir.
04

FastAPI nədir və rolu nədir?

FastAPI — Python üzərində müasir veb-çərçivədir. Verilənlər bazası deyil; MQTT-dən gələn xam mesajlar ilə PostgreSQL arasında etibarlı vasitəçi rolunu oynayır.

FastAPI nə edirNiyə birbaşa yazmırıq
Gələn məlumatı yoxlayır (temperatur ağlabatan aralıqdadırmı?)Xarab sensor mənfi çəki və ya 999°C göndərə bilər — bu, bazanı zibilləyər
Cihazın qeydiyyatdan keçib-keçmədiyini yoxlayırNaməlum cihazdan gələn məlumat sistemi çaşdıra bilər
Anomaliya aşkarlansa dərhal reaksiya verirSadə INSERT skripti bunu edə bilməz
Avtomatik API sənədləşməsi yaradır (Swagger)Gələcək inteqrasiyalar (dashboard, mobil) üçün hazır interfeys
JWT/API Key ilə təhlükəsizlik təmin edirDərs 10-da tam işlənəcək
Miqyas qeydi
Orijinal sənəddə FastAPI-nin Kafka-ya hadisə göndərməsi və çoxsaylı AI agentləri (Inventory, Production, Finance Agent) işə salması da göstərilir. Bu, tam müəssisə səviyyəsində, bir neçə zavod və modul arasında hadisə paylaşımı üçün nəzərdə tutulub. Bizim Faza 2-də tək bir zavodun sensor axını üçün bu, artıq mürəkkəblikdir — Kafka və çoxagentli aşkarlama sxemini **Faza 3**-ə saxlayırıq. Hazırda FastAPI birbaşa zavod_edge_db-yə yazır və xəbərdarlıq funksiyasını özü icra edir.
